## Шаг 3: OCR-сервис (2026-05-28)

### Что сделано:
- Создан `ocr_service.py` — модуль работы с OCR.space API:
  - `ocr_from_file(file_path)` — распознавание из файла на диске
  - `ocr_from_bytes(file_bytes, filename)` — распознавание из байтов (для Telegram)
  - Отправка изображений через `base64Image` (меньше проблем с encoding)
  - Отправка PDF через `multipart/form-data` (требуется для PDF)
  - Параметры OCR: `language=pol`, `isTable=true`, `scale=true`, `OCREngine=2`, `detectOrientation=true`
  - Парсинг ответа: поддержка многостраничных PDF, объединение текста со всех страниц
  - Обработка ошибок: HTTP-ошибки, ошибки API, пустой текст, некорректные exit-коды
  - Таймаут запроса: 120 секунд (для больших файлов)

### Файлы:
- `ocr_service.py`

## Шаг 4: Парсер текста фактур (2026-05-28)

### Что сделано:
- Создан `text_parser.py` — модуль для анализа распознанного OCR-текста:
  - Реализованы структуры данных `InvoiceItem` и `InvoiceData` для удобного хранения информации о товарах и фактуре.
  - Реализована функция `parse_invoice_text()` для извлечения даты, номера фактуры и товаров.
  - Извлечение даты покупки: регулярные выражения с поддержкой польских ключевых слов (`data sprzedaży`, `data wystawienia` и др.) и автонормализацией в формат `DD.MM.YYYY`.
  - Извлечение номера фактуры: поиск типичных номеров фактур (`Faktura VAT nr`, `Nr.`, `FV/...`, `FA/...`).
  - Парсинг товаров:
    - Стратегия 1: Поиск таблицы товаров по ключевым польским заголовкам (`nazwa`, `ilość`, `cena`, `wartość`, `opis`).
    - Стратегия 2: Если таблица не определилась, применяется гибкий построчный парсинг, исключающий технические блоки (NIP, REGON, адреса).
    - Очистка названия товара от номеров, ставок НДС (VAT) и единиц измерения (`szt`, `kg`, `opak` и др.).
    - Определение количества, цены за штуку и суммы с помощью анализа числовых комбинаций.
  - Расчет новых цен с учетом налога и маржи по формуле: `unit_price + (unit_price * tax%) + (unit_price * margin%)`.

### Файлы:
- `text_parser.py`

## Шаг 5: Google Sheets сервис (2026-05-28)

### Что сделано:
- Создан `sheets_service.py` — модуль для работы с Google Таблицами через API:
  - Настроена авторизация по сервисному аккаунту с использованием областей видимости `spreadsheets` и `drive`.
  - Реализована функция `get_or_create_spreadsheet()`, которая пытается открыть существующую таблицу по ID или создает новую с именем `Faktura Bot — Товары`, если ID отсутствует или некорректен.
  - Добавлена автоматическая настройка публичного доступа ("anyone with link can read") через Google Drive API (`insert_permission`) при создании таблицы, чтобы пользователь мог сразу открыть её.
  - При создании таблицы автоматически создаются и форматируются заголовки (первая строчка делается жирной).
  - Реализована функция `append_invoice_to_sheet()` для добавления товаров:
    - Рассчитываются новые цены с налогами и наценками.
    - Данные форматируются в виде плоских строк с поддержкой `USER_ENTERED` (чтобы числа правильно парсились самой таблицей).
    - Обрабатываются пропуски в дате и номере фактуры (заменяются на дефолтные прочерки `-`).
    - Пакетное добавление строк через `append_rows()` для оптимизации скорости работы и лимитов API.

### Файлы:

## Шаг 6: Хранение настроек пользователей (2026-05-28)

### Что сделано:
- Создан `user_settings.py` — менеджер настроек пользователей:
  - Реализован класс `UserSettingsManager`, работающий с локальным файлом `user_settings.json`.
  - У каждого пользователя теперь индивидуальные настройки:
    - `tax_percent` (налог, по умолчанию 5%)
    - `margin_percent` (маржа, по умолчанию 40%)
    - `spreadsheet_id` и `spreadsheet_url` (персональная Google Таблица). Благодаря этому пользователи бота не мешают друг другу и у каждого своя таблица.
  - Реализованы безопасные геттеры и сеттеры для всех параметров с автоинициализацией дефолтными значениями из конфигурации.
  - Создан глобальный объект `settings_manager` для простого импорта в других частях программы.
